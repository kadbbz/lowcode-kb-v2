# 低代码技术原理文章行文风格与格式指南

本文档规范了低代码技术原理系列文章的行文风格、格式要求和写作技巧，供后续写作参考。

---

## 一、总体风格定位

### 1.1 基调

工程实践视角，技术深度与可读性平衡：
- 不是纯学术论文，而是面向技术决策者的工程分析
- 不是入门教程，而是深度技术原理剖析
- 不是产品宣传，而是客观技术评估

### 1.2 态度

理性、批判、务实：
- 承认限制，不夸大能力
- 指出误区，不回避问题
- 强调工程可控性，不追求技术炫技

---

## 二、论述结构模式

### 2.1 典型章节结构

```
第X章 主标题

开篇段落：背景定位、核心价值、章节概述

## 一、第一个大节
### X.X 子标题：核心机制/价值
开篇段落：这个机制的目的和价值
具体案例：订单系统等场景化说明
分点展开：3-4个要点，层层递进
小结：机制的价值和意义

## 二、第二个大节
（同上结构）

## N、最后总结
回到整体价值，提升到更高层次
```

### 2.2 每个小节的写作节奏

1. 主题句：点明本节核心观点
2. 具体场景：通过案例说明（订单、审批等）
3. 分层展开：用列表或段落详细阐述
4. 对比分析：与传统方案或错误做法对比
5. 总结升华：回到工程价值和原则

## 三、语言技巧

### 3.1 类比手法（大量使用，增强可理解性）

技术类比：
- SQL的schema定义
- Java的向后兼容性
- Git的版本管理
- 虚拟机的字节码执行
- 操作系统的进程管理
- React的JSX
- Excel的公式引擎
- BPMN的流程定义

生活类比：
- 新能源汽车改变出行成本结构

### 3.2 案例选择（具体且连贯）

主要案例系列（订单管理系统）：
- 订单详情页面（页面级案例）
- 客户名称输入框（组件级案例）
- 订单审批流程（流程级案例）
- 字段重命名（级联更新案例）
- VIP折扣计算（代码风格差异案例）

案例使用原则：
- 围绕订单系统等核心案例，保持系列连贯性
- 在不同章节中继续使用相同的业务背景
- 避免每个案例都是新的，导致读者无法建立连续认知

### 3.3 术语处理

基本规范：
- 首次出现给出英文和定义：如设计器（Designer）
- 专业术语用中文表达：元数据、运行时、依赖关系
- 关键概念适度强调，但避免过度使用粗体
- 代码和命令保持英文：`eval()`、`window`、`customerName`

常用术语对照表：

| 中文术语 | 英文术语 | 使用规范 |
|---------|---------|---------|
| 元数据 | Metadata | 首次出现可带英文 |
| 设计器 | Designer | 首次出现可带英文 |
| 运行时 | Runtime | 首次出现可带英文 |
| 领域专用语言 | DSL | 首次出现全称加缩写 |
| 数据定义语言 | DDL | 首次出现全称加缩写 |
| 基于角色的访问控制 | RBAC | 首次出现全称加缩写 |
| 增删改查 | CRUD | 可直接使用CRUD |
| 企业资源计划 | ERP | 可直接使用ERP |
| 商业智能 | BI | 可直接使用BI |

### 3.4 对比表格（清晰对比不同方案）

标准格式：
```
| 维度 | × 不足方案 | × 过度方案 | √ 平衡方案 |
|------|-----------|-----------|----------|
| 条件表达式 | 只支持简单比较 | 支持任意JavaScript | 受限的表达式DSL |
| 数据查询 | 只能获取当前数据 | 可以执行任意SQL查询 | 显式声明的实体和查询规则 |
| 扩展机制 | 完全封闭 | 可以注入任意代码 | 受控的插件接口 |
```

### 3.5 强调方式（减少粗体使用）

基本原则：
- 关键术语首次出现时可以加粗
- 核心概念可以使用引号强调
- 优先使用逻辑结构和段落层次来突出重点
- 列表和分层展示代替过度加粗

具体规范：
- 避免大量使用粗体强调，导致重点不突出
- 同一段落中粗体不超过3处
- 连续段落中避免每段都有粗体
- 小标题本身已经是强调，不要在小标题后的第一句再加粗

## 四、段落与句式特征

### 4.1 段落长度

开篇段落：100-150字，定位整节内容
展开段落：150-250字，阐述一个完整观点
列表项段落：80-150字，聚焦一个要点
总结段落：80-120字，提炼核心价值

### 4.2 句式特点

定位句式：如果说A是X，那么B就是Y
- 例：如果说设计器是生产车间，那么运行时就是执行引擎
- 使用场景：章节开篇、大节开头、需要建立类比关系时

递进句式：不仅是X，更重要的是Y
- 例：运行时的职责不仅是能够执行，更重要的是保证执行的一致性
- 使用场景：强调核心价值、突出关键差异

对比句式：传统方案X，而元数据驱动Y
- 例：传统代码在编译时才能发现错误，而设计器通过实时验证将错误前移到设计时
- 使用场景：对比分析、突出优势

类比句式：这类似于A中的B
- 例：这种约束机制类似于虚拟机对字节码的安全约束
- 使用场景：解释复杂概念、增强可理解性

强调句式：正是X才Y、这正是Z
- 例：这正是元数据驱动相比代码生成方案的重要优势
- 使用场景：总结升华、强化结论

引用句式：用于突出核心观点
- 例：软件系统的复杂度被长期分散在大量命令式代码和个人决策中，缺乏可被平台统一理解、治理和演进的表达形式
- 使用场景：提出核心论断、引出关键问题
- 格式：使用引号包裹，独立成段或放在段落开头

### 4.3 高频句式积累

定位句式：
- 如果说A是X，那么B就是Y
- X是Y的核心环节
- X并非可有可无的辅助工具，而是
- 理解X的工作机制，是理解Y的关键

职责句式：
- X的首要职责，是
- X的核心职责，不仅是A，更重要的是B
- X必须、需要、应该
- X负责、承担、保证

价值句式：
- 这正是X相比Y的重要优势
- X使得Y成为可能
- 通过X，能够实现Y
- X的价值在于Y

对比句式：
- 传统方案X，而元数据驱动Y
- 如果不X，将会Y
- 相比X，Y的优势在于Z
- X虽然能够A，但B；而Y通过C，实现了D

因果句式：
- 正是因为X，Y才
- 由于X，Y必须
- 这种X，确保或避免了Y
- X导致、带来、引发Y

类比句式：
- 这类似于X中的Y
- X接近于Y
- X可以被看作是Y
- 这一要求类似于X

## 五、逻辑推进模式

### 5.1 总-分-总结构

总：章节开篇定位，点明核心职责
分：3-5个子标题，每个聚焦一个维度
总：最后一个子标题，回到整体价值

### 5.2 层层递进

1. 是什么：定义和定位
2. 为什么：为何重要、解决什么问题
3. 怎么做：具体机制和实现方式
4. 达到什么效果：价值和优势
5. 与其他部分的关系：整体协同

### 5.3 正反对比

先说反面：如果不这样做，会有什么问题
再说正面：通过这种机制，能够解决什么
强化对比：相比传统方案的优势

---

## 六、具体写作指导

### 6.1 章节开篇模板

```
# 第X章 [章节主题]

[开篇段落，包含以下要素：]
- 背景引入：从前一章或现实问题引入
- 核心问题：本章要回答的关键问题
- 价值定位：为什么这个问题重要
- 章节概述：本章将讨论哪些方面

## 一、第一个大节标题

### X.X [小节标题]

[主题句：这个机制的目的]
[具体场景或案例]
[分层展开]
[总结升华]
```

### 6.2 小节展开模板

```
### X.X [小节标题]：[核心机制/价值]

[主题句：这个机制的目的]

[场景引入]
以[具体案例]为例。当[场景描述]时，[组件]需要完成以下步骤：

第一步或第一点：[内容]。例如，[具体细节]。这种[机制]类似于[类比]，但[差异]。

第二步或第二点：[内容]。例如，[具体细节]。如果[反面情况]，将导致[问题]。

[过渡句]
这个过程的关键在于[核心概念]。

[总结段落]
通过这些[机制名称]，[组件]能够[达到的效果]。它确保了[价值1]、[价值2]、[价值3]，避免了[避免的问题]。
```

### 6.3 案例描述规范

好的案例描述：
- 具体：订单详情页面、customerName字段、提交审批按钮
- 场景化：当用户点击时、当开发者配置时
- 分步骤：首先、然后、最后
- 有细节：宽度200px、类型为input、绑定到orderData.customerName

避免的案例描述：
- 抽象：某个页面、某个字段、某个按钮
- 泛泛而谈：系统会处理、平台会执行
- 跳步骤：直接说结果，不说过程
- 无细节：只说配置，不说具体值

### 6.4 对比分析模板

三方对比（元数据驱动 vs 传统方案1 vs 传统方案2）：
```
[机制名称]，使得元数据驱动相比[传统方案]的重要优势：[优势1]和[优势2]。

- 传统[方案1]：[问题描述]，导致[后果]
- 传统[方案2]：虽然[优点]，但[缺点]
- 元数据驱动：通过[机制]，实现[效果]，同时[避免问题]
```

时间维度对比（设计时 vs 运行时）：
```
传统代码开发模式下，[错误类型]只有在[时机]才能被发现，此时的[成本]已经较高。而[组件]通过[机制]，将[错误]拦截在[时机]，开发者可以立即修正，避免[问题]。
```

### 6.5 技术深度把控

适中的深度（当前文章的水平）：
- 说明元数据包含组件类型、属性、数据绑定、验证规则
- 举例：readonly属性为false、width为200px、绑定到orderData.customerName
- 类比：类似于SQL的schema定义、Java的向后兼容
- 后果：如果不验证，会导致运行时错误，修复成本高

过浅（应避免）：
- 只说元数据定义了页面，不说包含哪些内容
- 只说设计器很重要，不说为什么重要
- 只说运行时执行元数据，不说如何执行

过深（应避免）：
- 深入到元数据的JSON结构细节
- 讨论具体的算法实现：依赖追踪用DAG还是邻接表
- 涉及具体的技术栈：React还是Vue、Redux还是MobX

## 七、标点符号规范

### 7.1 必须使用中文标点（除代码外）

基本标点：
- 逗号：，（不是,）
- 句号：。（不是.）
- 冒号：：（不是:）
- 分号：；（不是;）

配对标点：
- 括号：（）（不是()）
- 引号：""（不是""或''）

特殊标点：
- 顿号：、（用于并列）
- 破折号：——（不是--）
- 省略号：……（不是...）

### 7.2 代码和技术术语保持英文

代码示例：
- 代码：`eval()`、`window.document`
- 字段名：customerName、orderData
- 类型名：input、boolean
- 技术缩写：JSON、XML、DSL、CRUD

使用场景：
- 行内代码块中的所有内容
- 代码示例块中的所有内容
- 引用具体的变量名、函数名、类名时
- 引用技术标准和规范的英文缩写时

## 八、常见写作误区

### 8.1 误区1：过度技术化

错误做法：
- 深入讨论JSON结构细节
- 讨论具体的算法实现
- 涉及具体的技术栈选择

正确做法：
- 聚焦工程原理
- 关注设计思路
- 进行价值分析

### 8.2 误区2：过于抽象

错误做法：
- 只说平台很强大、设计很重要
- 泛泛而谈系统能力
- 缺乏具体场景和细节

正确做法：
- 具体说明强大在哪里
- 解释为什么重要
- 描述如何实现

### 8.3 误区3：缺乏案例

错误做法：
- 只有理论阐述
- 没有具体场景
- 观点缺乏支撑

正确做法：
- 每个观点都有对应的案例
- 场景化说明机制
- 用实例支撑观点

### 8.4 误区4：案例不连贯

错误做法：
- 每个案例都是新的
- 业务背景频繁切换
- 读者无法建立连续认知

正确做法：
- 围绕订单系统等核心案例
- 保持业务背景一致
- 展开系列场景

### 8.5 误区5：忽视对比

错误做法：
- 只说元数据驱动如何
- 不说相比其他方案的优势
- 缺乏横向比较

正确做法：
- 通过对比突出独特价值
- 说明相对优势
- 指出其他方案的局限

### 8.6 误区6：过度使用粗体

错误做法：
- 大量使用粗体强调
- 每段都有多处粗体
- 导致重点不突出

正确做法：
- 适度使用粗体
- 主要用于首次出现的核心术语
- 优先使用结构和层次突出重点

## 九、质量检查清单

### 9.1 内容完整性检查

- 是否有明确的定位段落
- 是否有具体的案例说明
- 是否有分层的机制阐述
- 是否有与其他部分的关联
- 是否有总结性段落

### 9.2 逻辑连贯性检查

- 段落之间有过渡句吗
- 观点之间有递进关系吗
- 案例与观点是否对应
- 前后文是否有矛盾
- 术语使用是否一致

### 9.3 技术准确性检查

- 技术概念定义准确吗
- 类比恰当且不误导吗
- 案例细节真实可信吗
- 对比分析客观公正吗
- 避免了过度承诺吗

### 9.4 可读性检查

- 段落长度适中吗（80-250字）
- 句子复杂度适中吗（避免超过3层嵌套）
- 专业术语有解释吗
- 使用了恰当的格式化吗（适度加粗、列表）
- 避免了过多的英文缩写吗

### 9.5 风格一致性检查

- 与已有章节的口吻一致吗
- 案例系列连贯吗（订单系统）
- 标点符号规范吗（中文标点）
- 标题层级正确吗
- 强调方式适度吗（避免过度加粗）

## 十、快速写作检查表

### 10.1 结构检查

- 有开篇定位段落
- 有3-5个大节
- 每个大节有清晰主题
- 有总结性段落

### 10.2 内容检查

- 有具体案例（最好是订单系统）
- 有类比说明
- 有对比分析
- 有价值升华

### 10.3 格式检查

- 标点符号全中文（除代码）
- 粗体使用适度，主要用于核心术语首次出现
- 列表格式统一
- 段落长度适中

### 10.4 风格检查

- 口吻与前文一致
- 术语使用统一
- 案例系列连贯
- 深度适中（不过浅不过深）

## 十一、后续写作建议

### 11.1 保持一致性

- 延续订单系统案例系列：在新场景中继续使用订单管理的例子
- 保持工程视角：继续从工程实践角度分析
- 保持批判性思维：客观指出问题和限制

### 11.2 深化关键概念

- 进一步展开工程可控性、长期演进等核心概念
- 补充更多对比：与代码生成、其他技术路线对比
- 探讨边界与限制：客观分析能力边界和适用场景

### 11.3 强化协同关系

- 进一步阐述元数据、设计器、运行时三者的协同机制
- 关注成本与价值：持续强调低代码如何改变企业软件的成本结构
- 保持价值导向：回归企业软件的真实需求和痛点

通过遵循本指南，后续章节可以保持与已完成章节的风格一致性和格式规范性。
